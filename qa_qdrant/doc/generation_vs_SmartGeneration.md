# Q/A生成方式 完全比較ガイド

**スマート生成（SmartQAGenerator）vs 従来方式（Legacy Mode）**

## 🎯 結論: スマート生成がデフォルト（推奨）

**現在のデフォルト**: `SmartQAGenerator`による**スマート生成モード**

v2.1以降、`make_qa_register_qdrant.py`はスマート生成をデフォルトで使用します。

```
make_qa_register_qdrant.py
    ↓
QAPipeline.run(use_smart_generation=True)  ← デフォルトはTrue
    ↓
QAPipeline.generate_qa()
    ↓
generate_qa_dataset(..., use_smart_generation=True)
    ↓
QAGenerator(..., use_smart_generation=True)
    ↓
QAGenerator._generate_smart()  ← スマート方式を使用
    ↓
SmartQAGenerator.process_chunk()
```

### なぜスマート生成がデフォルトなのか？

| 観点 | スマート生成の優位性 |
|------|---------------------|
| **品質** | LLMによる内容理解で高品質なQ/Aを生成 |
| **効率** | 不要なチャンクは0個生成でスキップ可能 |
| **カバレッジ** | 主要トピックを確実にカバー |
| **メタデータ** | topic, importance_score, complexity付きで検索・フィルタリングに有利 |
| **適応性** | チャンクの特性に応じた柔軟なQ/A数（0-5個） |

## 2つの方式の概要

### スマート生成（Smart Mode）- **デフォルト・推奨**

| 項目 | 内容 |
|------|------|
| **モジュール** | `qa_generation/smart_qa_generator.py` |
| **クラス** | `SmartQAGenerator` |
| **アプローチ** | LLMによるチャンク分析 → 動的Q/A数決定 → 生成 |
| **LLM呼び出し** | 2回/チャンク（分析 + 生成） |
| **Q/A数範囲** | 0-5個（柔軟） |
| **特徴** | インテリジェント、高品質、トピック明示 |

### 従来方式（Legacy Mode）- オプション

| 項目 | 内容 |
|------|------|
| **モジュール** | `qa_generation/generation.py` |
| **クラス** | `QAGenerator` (legacy mode) |
| **アプローチ** | トークン数カウント → 固定ルールでQ/A数決定 → 生成 |
| **LLM呼び出し** | 1回/チャンク |
| **Q/A数範囲** | 2-8個（固定ルール） |
| **特徴** | 高速、低コスト、シンプル |

## スマート生成の詳細（Smart Mode）

### 処理フロー

| ステップ | 処理内容 | 詳細 |
|---------|---------|------|
| **入力** | チャンク | テキストデータ |
| **ステップ1** | チャンク分析（LLMによる） | 情報密度評価、重要度スコア計算（0.0-1.0）、複雑さ判定（low/medium/high）、主要トピック抽出、適切なQ/A数決定（0-5個） |
| **ステップ2** | Q/A生成（LLMによる） | 分析結果に基づく動的プロンプト、主要トピックの明示的指示、重要度に応じた品質指示、トピック付きQ/A生成 |
| **ステップ3** | 品質チェック | 期待Q/A数との整合性確認、トピックフィールドの補完、重複チェック |
| **出力** | Q/Aペアリスト（動的数）+ 分析メタデータ | |

### Q/A数決定の判断基準

| Q/A数 | 判断条件 | 例 |
|-------|---------|---|
| **0個** | メタ情報のみ、補足情報、意味のない繰り返し | 「詳細は付録参照」「ページ番号」 |
| **1個** | 単一事実の記述 | 単純な定義文 |
| **2個** | 関連する2つの事実 | 製品の色とサイズ |
| **3個** | 標準的な説明パラグラフ | 複数の関連情報 |
| **4-5個** | 高密度技術情報、複数の独立したポイント | 暗号化アルゴリズムの詳細 |

### チャンク分析の出力

```python
{
    'qa_count': int,           # 生成すべきQ/A数（0-5）
    'key_topics': List[str],   # 主要トピック
    'importance_score': float, # 重要度（0.0-1.0）
    'complexity': str,         # 複雑さ（low/medium/high）
    'reasoning': str           # 判断理由
}
```

### 出力データ構造

```python
qa_pair = {
    "question": "質問文",
    "answer": "回答文",
    "question_type": "fact",  # デフォルト
    "topic": "暗号化方式",    # ✨ スマート生成固有
    "source_chunk_id": "chunk_0",
    "doc_id": "doc_0",
    "dataset_type": "wikipedia_ja",
    "chunk_idx": 0,
    # ✨ スマート生成メタデータ
    "generation_method": "smart",
    "importance_score": 0.85,
    "complexity": "high"
}
```

### スマート生成の長所と短所

| 長所 ✅ | 短所 ❌ |
|--------|--------|
| インテリジェント（内容を理解してQ/A数決定） | 低速（LLM呼び出し2回必要、処理時間が約2倍） |
| 効率的（不要なQ/A生成を回避、0個生成も可能） | 高コスト（APIコストが約2倍） |
| 高品質（主要トピックを確実にカバー、重要情報を優先） | 複雑性（デバッグが困難、エラーハンドリングが複雑） |
| 豊富なメタデータ（トピックラベル、重要度スコア、複雑さ情報） | バッチ処理非対応（チャンクごとに分析が必要） |
| 適応的（チャンクの特性に応じて変化、柔軟なQ/A数0-5個） | 不確実性（LLM判断に依存、分析結果のばらつき） |

## 従来方式の詳細（Legacy Mode）

### 処理フロー

| ステップ | 処理内容 | 詳細 |
|---------|---------|------|
| **入力** | チャンク | テキストデータ |
| **ステップ1** | Q/A数の決定 | トークン数カウント、固定的なルールベース計算、位置バイアス補正 |
| **ステップ2** | プロンプト生成 | システムプロンプト（教育コンテンツ専門家）、ユーザープロンプト（Q/A数固定指示）、質問タイプ指定、JSON形式での出力指示 |
| **ステップ3** | LLM API呼び出し | generate_structured() - 構造化出力 |
| **ステップ4** | 後処理 | Pydanticモデルでパース、メタデータ追加、リスト化 |
| **出力** | Q/Aペアリスト（固定数） | |

### Q/A数決定ロジック

| トークン数 | Q/A数 |
|-----------|-------|
| < 50 | 2個 |
| < 100 | 3個 |
| < 200 | base_count + 1 (通常4個) |
| < 300 | base_count + 2 (通常5個) |
| >= 300 | base_count + 3 (通常6個) |
| 後半チャンク（位置 >= 5） | +1個 |
| **最大値** | 8個 |

```python
def _legacy_determine_qa_count(self, chunk: Dict, config: Dict) -> int:
    """従来方式のQ/A数決定（後方互換性）"""
    base_count = config["qa_per_chunk"]  # 通常は3
    token_count = self.client.count_tokens(chunk['text'], model=self.model)
    chunk_position = chunk.get('chunk_idx', 0)

    if token_count < 50:
        qa_count = 2
    elif token_count < 100:
        qa_count = 3
    elif token_count < 200:
        qa_count = base_count + 1
    elif token_count < 300:
        qa_count = base_count + 2
    else:
        qa_count = base_count + 3

    if isinstance(chunk_position, int) and chunk_position >= 5:
        qa_count += 1

    return min(qa_count, 8)
```

### 出力データ構造

```python
qa_pair = {
    "question": "質問文",
    "answer": "回答文",
    "question_type": "fact",  # fact/reason/comparison/application
    "source_chunk_id": "chunk_0",
    "doc_id": "doc_0",
    "dataset_type": "wikipedia_ja",
    "chunk_idx": 0,
    "generation_method": "legacy"  # 従来方式の印
}
```

### 従来方式の長所と短所

| 長所 ✅ | 短所 ❌ |
|--------|--------|
| 高速処理（LLM呼び出し1回のみ、トークンカウントのみで計算） | 非効率な生成（不要なチャンクからもQ/A生成、情報量に対して過剰/不足の可能性） |
| 低コスト（APIコストが最小、大量データ処理に適している） | 品質のばらつき（重要度を考慮しない、トピックカバレッジの保証なし） |
| 安定性（シンプルなロジック、デバッグが容易） | 冗長性（低品質なQ/Aも生成される、後処理フィルタリングが必要） |
| 後方互換性（既存システムと完全互換） | メタデータ不足（トピック情報なし、重要度スコアなし） |
| バッチ処理対応（複数チャンクを一度に処理可能） | |

```bash

python make_qa_register_qdrant.py \
  --input-file document.txt \
  --collection my_docs \
  --recreate
```

## まとめ

### Q1: `make_qa_register_qdrant.py`が使用しているのは？

**答え**: `qa_generation/smart_qa_generator.py`の**スマート生成モード（Smart Mode）**（デフォルト）

- `use_smart_generation=True`がデフォルト
- LLMによる内容分析でQ/A数を動的決定
- 2回のLLM呼び出しで高品質なQ/Aを生成

### Q2: 2つの方式の最大の違いは？

| 項目 | スマート生成 ⭐デフォルト | 従来方式 |
|------|------------------------|---------|
| **Q/A数決定** | LLMによる内容分析 | トークン数のみ |
| **処理速度** | 低速（2倍） | 高速 |
| **品質** | ⭐⭐⭐ 高品質 | ⭐⭐ 標準 |
| **コスト** | 高い（2倍） | 低い |
| **0個生成** | ✅ 可能 | ❌ 不可 |
| **メタデータ** | 豊富（topic, importance, complexity） | 基本のみ |

### Q3: どちらを使うべきか？

| ケース | 推奨方式 |
|-------|---------|
| **品質重視、少〜中量データ、メタデータ活用** | ⭐ **スマート生成（デフォルト）** |
| **大規模処理、速度重視、コスト重視** | 従来方式（`--no-smart-generation`） |

### スマート生成がデフォルトである理由

1. **品質が重要** - RAGシステムの検索精度はQ/Aの品質に依存
2. **効率的** - 不要なQ/A生成を回避し、ストレージ・検索効率が向上
3. **メタデータ活用** - topic, importance_scoreでフィルタリング・ランキングが可能
4. **主要トピックカバレッジ** - 重要な情報を漏らさない
